# Track Loading - Explanation ## Where Tracks Are Coming From ### Issue: Tracks Appearing in Console The tracks you saw in the console (BAQ, CD, DMR, GP, LRL, MNR) were coming from: 1. **Old System**: `AppContext` `loadAndMergeAllTracks()` Hardcoded tracks Old backend API Upstash cache 2. **Location**: `lib/ingest.ts` line 380 had hardcoded tracks: `["BAQ", "CD", "DMR", "GP", "LRL", "MNR"]` 3. **Trigger**: `AppContext` was calling `loadData()` automatically on page load when `connections.length === 0` ### Why They Appeared After Clearing Upstash - Even after clearing Upstash, the old system was still fetching from MongoDB via the old backend API - The old API (`/api/tracks/{track}/entries`) caches data in Upstash - So tracks would reappear in Upstash after the automatic load ### Why They Weren't in Supabase - The old system stores data in **Upstash (Redis)**, not Supabase - Only the **new system** (admin panel MongoDB fetch) stores data in Supabase `track_data` table ## Fix Applied ### 1. Disabled Automatic Track Loading - Modified `loadAndMergeAllTracks()` to return empty array if no track is selected - Modified `AppContext` to not auto-load on admin page - Tracks now only load when: - User explicitly selects a track on matchups page - Admin fetches track data via MongoDB endpoint ### 2. Admin Page Protection - `AppContext` now checks if we're on `/admin` page - Skips automatic loading on admin page - Prevents unnecessary data fetching ## New System Flow ### Admin Fetch (Correct Flow) 1. Admin enters track code + date 2. Clicks "Fetch Track Data from MongoDB" 3. Frontend calls: `POST /api/admin/tracks/fetch` 4. Backend: - Fetches from MongoDB - Transforms data - Stores in Supabase `track_data` table - Caches in Redis (for performance) - Returns preview 5. Admin sees preview, creates contest 6. Contest uses `track_data_id` to reference the data ### Old System (Now Disabled) 1. `AppContext` auto-loads on page load 2. Loads hardcoded tracks from old API 3. Caches in Upstash 4. **This is what was causing the issue** ## Testing After the fix: 1. No tracks should appear in console automatically 2. No tracks should appear in Upstash automatically 3. Admin page should not trigger track loading 4. "Fetch Track Data" button should work (from MongoDB) 5. Data should appear in Supabase `track_data` table ## Next Steps 1. **Clear Upstash** (if needed): ```bash npx tsx scripts/clear-redis.ts ``` 2. **Refresh admin page**: - Should see no track loading logs - Should see no data in Upstash 3. **Test Fetch Button**: - Enter track + date - Click "Fetch Track Data" - Should see MongoDB fetch logs - Should see preview - Should see data in Supabase
