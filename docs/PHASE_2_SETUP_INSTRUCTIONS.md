# Phase 2: Database Schema Setup Instructions ## Overview Phase 2 adds the database schema needed for the new contest creation flow. This includes: - `track_data` table for storing MongoDB data and metadata - `matchups` table for storing calculated matchups - `operations` table for tracking async operations - Updates to `contests` table (new columns for status, matchup types, etc.) ## Prerequisites - Supabase project set up - Phase 1 fixes completed - Admin access working ## Setup Steps ### Step 1: Run the SQL Migration 1. Open your Supabase project dashboard 2. Go to **SQL Editor** 3. Click **New Query** 4. Copy the entire contents of `docs/PHASE_2_DATABASE_SCHEMA.sql` 5. Paste into the SQL Editor 6. Click **Run** (or press Cmd/Ctrl + Enter) ### Step 2: Verify Tables Were Created Run this query in SQL Editor to verify: ```sql SELECT table_name FROM information_schema.tables WHERE table_schema = 'public' AND table_name IN ('track_data', 'matchups', 'operations') ORDER BY table_name; ``` You should see: - `track_data` - `matchups` - `operations` ### Step 3: Verify Contests Table Updates Run this query to verify new columns were added: ```sql SELECT column_name, data_type, column_default FROM information_schema.columns WHERE table_name = 'contests' AND column_name IN ('status', 'track_data_id', 'matchup_types', 'matchups_calculated_at') ORDER BY column_name; ``` You should see all 4 columns: - `status` (TEXT, default: 'draft') - `track_data_id` (UUID, nullable) - `matchup_types` (TEXT[], default: empty array) - `matchups_calculated_at` (TIMESTAMPTZ, nullable) ### Step 4: Verify RLS Policies Run this query to verify RLS is enabled: ```sql SELECT tablename, rowsecurity FROM pg_tables WHERE schemaname = 'public' AND tablename IN ('track_data', 'matchups', 'operations') ORDER BY tablename; ``` All should show `rowsecurity = true`. ### Step 5: Test Admin Access 1. Go to your admin panel in the app 2. Check that existing contests still show correctly 3. Verify no errors in browser console ## What Changed ### New Tables #### `track_data` - Stores full MongoDB data for each track/date - Includes metadata (race count, horse count, post times) - Unique constraint on (track_code, date) #### `matchups` - Stores calculated matchups for contests - One row per contest + matchup_type combination - Links to contests via `contest_id` #### `operations` - Tracks async operations (fetching data, calculating matchups) - Includes progress tracking (0-100%) - Stores input/output data and error messages ### Updated `contests` Table New columns: - `status`: 'draft', 'processing', 'ready', 'active' - `track_data_id`: Links to track_data table - `matchup_types`: Array of matchup types ['jockey_vs_jockey', etc.] - `matchups_calculated_at`: Timestamp when matchups were calculated ### Existing Data - **Existing contests** are automatically set to `status = 'ready'` (they were created before this system) - **No data loss** - all existing contests remain intact - **Backward compatible** - existing code continues to work ## Rollback (If Needed) If you need to rollback, run this (️ **WARNING: This will delete new tables**): ```sql -- Drop new tables (CASCADE will also drop related objects) DROP TABLE IF EXISTS matchups CASCADE; DROP TABLE IF EXISTS operations CASCADE; DROP TABLE IF EXISTS track_data CASCADE; -- Remove new columns from contests (if needed) ALTER TABLE contests DROP COLUMN IF EXISTS status, DROP COLUMN IF EXISTS track_data_id, DROP COLUMN IF EXISTS matchup_types, DROP COLUMN IF EXISTS matchups_calculated_at; ``` ## Next Steps After Phase 2 is complete: 1. Verify all tables exist 2. Verify RLS policies are active 3. Test admin panel still works 4. **Ready for Phase 3: Backend APIs** ## Troubleshooting ### "Table already exists" error - This is safe to ignore - the script uses `CREATE TABLE IF NOT EXISTS` - The script is idempotent (safe to run multiple times) ### "Column already exists" error - This is also safe to ignore - columns are created with `DO $$` blocks that check first - The script is idempotent ### RLS policy errors - If you see policy errors, check that the `profiles` table exists and has the `is_admin` column - Make sure `is_user_admin()` function exists (from Phase 1) ### Foreign key constraint errors - If you see these, make sure the `contests` table exists first - The script should handle this automatically ## Questions? If you encounter any issues: 1. Check the Supabase logs (Dashboard Logs) 2. Verify all prerequisites are met 3. Check browser console for frontend errors 4. Verify admin access is working
