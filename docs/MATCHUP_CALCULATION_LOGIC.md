# Matchup Calculation Logic ## Overview Matchups are calculated from track entries (not results), but only count stats from horses that actually raced. ## Key Principles 1. **Connections (Jockeys, Trainers, Sires)**: Counted from ALL entries - Includes scratched horses (they still have connections) - Includes Also Eligible (AE) horses (they still have connections) - This is why metadata shows jockeys/trainers/sires from all records 2. **Stats (Apps, Points, Salary)**: Only counted from horses that actually raced - Excludes scratched horses - Excludes AE horses (they didn't race) - Only includes horses with `place !== null` (finished the race) 3. **Results Data (Points, Place)**: Only used on Results page - Matchups page: Shows connections, salaries, apps (from entries) - Results page: Shows points, finishing positions, Points/1K$ (from results) ## How It Works ### 1. Track Data Fetching (`admin-track-data.ts`) - Fetches entries from MongoDB - Fetches results from MongoDB - Merges them to calculate points/place - Stores merged data in Supabase ### 2. Metadata Calculation (`trackDataTransform.ts`) - **Total Horses**: All entries (including scratched and AE) - **Non-Scratched**: Excludes scratched, includes AE - **Jockeys/Trainers/Sires**: Counted from ALL entries (connections are valid even if horse didn't race) ### 3. Matchup Calculation (`matchupCalculation.ts`) - Uses merged records (entries + results) - Generates connections for each matchup type (JvJ, TvT, SvS) - Only counts apps/points from `actuallyRaced` horses: ```typescript const actuallyRaced = !isScratched && !isAlsoEligible && record.place !== null; ``` - Uses unified generator logic (same as Python script) ### 4. Frontend Display - **Matchups Page**: Shows connections, salaries, apps (no results data) - **Results Page**: Shows points, finishing positions, Points/1K$ (from results) ## Data Flow ``` MongoDB Entries Transform Merge with Results Store in Supabase Calculate Metadata Generate Connections Calculate Matchups Store in Supabase Frontend: Matchups Page (no results) Frontend: Results Page (with results) ``` ## Why This Approach? 1. **Connections are valid even if horse didn't race** - A jockey/trainer/sire connection exists regardless of whether the horse scratched or was AE 2. **Stats only count actual races** - We only want to compare connections based on horses that actually competed 3. **Results separate from matchups** - Users shouldn't see results when making picks, only when viewing outcomes
