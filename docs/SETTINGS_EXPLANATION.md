# Matchup Calculation Settings Explanation ## How Settings Work ### The Pre-Generation Strategy When you set the **"Count"** in the matchup settings, you're specifying **how many matchups you want users to SEE**, not how many to generate. Here's how it works: 1. **Admin Sets Count** (e.g., 10 for JvJ) - This is the number of matchups that will be displayed to users 2. **Backend Generates Pool** (automatically 3x the count) - If you set 10, the backend generates **30 matchups** and stores them in the database - This creates a pool of matchups for instant reloads 3. **Frontend Randomly Selects** (from the pool) - When a user loads the matchups page, they see **10 random matchups** from the pool of 30 - When they click "Reload", they get a **different random set of 10** from the same pool - This provides instant reloads without recalculating ### Recommended Settings Based on user preferences and connection popularity: #### Jockey vs Jockey (JvJ) - **Count: 10** (shows 10, generates 30) - **Prefer 1v1: 0.9** (90% chance of 1v1 matchups) - **Tolerance: 300** - **Why**: Jockeys are the most recognizable connections. Users prefer simple 1v1 matchups. #### Trainer vs Trainer (TvT) - **Count: 10** (shows 10, generates 30) - **Prefer 1v1: 0.9** (90% chance of 1v1 matchups) - **Tolerance: 300** - **Why**: Trainers are also well-known. 1v1 matchups are easier to understand and compare. #### Sire vs Sire (SvS) - **Count: 8** (shows 8, generates 24) - **Prefer 1v1: 0.8** (80% chance of 1v1 matchups) - **Tolerance: 300** - **Why**: Sires are less familiar to casual users, but still benefit from 1v1 simplicity. #### Mixed - **Count: 15** (shows 15, generates 45) - **Prefer 1v1: 0.7** (70% chance of 1v1 matchups) - **Tolerance: 300** - **Why**: Mixed matchups benefit from variety (2v1, 1v2 patterns) to showcase different connection types. ### Settings Parameters - **Count**: Number of matchups to show to users (backend generates 3x this) - **Tolerance**: Salary difference allowed between sets (300 is standard) - **Prefer 1v1**: Probability of generating 1v1 vs multi-connection matchups (0-1 scale) - **Max Attempts**: Maximum tries to find valid matchups (500 is standard) ### Why 1v1 for Jockeys and Trainers? 1. **Familiarity**: Users know jockey and trainer names better than sires 2. **Simplicity**: 1v1 is easier to understand and compare 3. **Engagement**: Simpler matchups = faster decisions = more engagement 4. **Balance**: Still allows some 2v1/1v2 for variety, but prioritizes 1v1 ### Example Flow 1. Admin sets JvJ count to **10** 2. Backend generates **30 JvJ matchups** and stores them 3. User loads matchups page sees **10 random matchups** from the 30 4. User clicks "Reload" sees **different 10 random matchups** from the same 30 5. After 15 reloads, user has seen all 30 matchups (with some repeats) This strategy provides: - Instant reloads (no recalculation needed) - Variety (different matchups on each reload) - Performance (pre-generated, cached in database) - User experience (fast, responsive)
