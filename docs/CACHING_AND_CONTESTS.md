# Caching & Contest Configuration System ## Overview This system implements industry-standard caching and contest management to: - **Reduce MongoDB calls** by caching entries/results data - **Enable pre-fetching** contests before they start (1 day before) - **Support admin configuration** of which tracks/dates to show to players - **Scale efficiently** without exhausting MongoDB credits ## Architecture ### 1. Redis Caching (Upstash) **Location**: `backend/src/utils/redis.ts` - Uses **Upstash Redis** (free tier available) - Falls back to in-memory cache if Redis not configured - **TTL Strategy**: - **Past data**: 24 hours (stable, won't change) - **Future data**: 1 hour (may update as race approaches) **Setup**: ```bash # In backend/.env UPSTASH_REDIS_REST_URL=https://your-redis-instance.upstash.io UPSTASH_REDIS_REST_TOKEN=your_redis_token ``` If Redis is not configured, the system automatically falls back to in-memory caching. ### 2. Contest Configuration **Location**: `backend/src/utils/contests.ts` **Current Implementation**: - Hardcoded contests for November 2, 2025 - 6 tracks: BAQ, CD, DMR, GP, LRL, MNR **Future Implementation** (Admin Panel): - Store contests in Supabase `contests` table - Admin can enable/disable contests - Support for future dates (pre-fetch data 1 day before) **API Endpoints**: - `GET /api/contests` - Get all active contests - `GET /api/contests/:track/:date` - Get specific contest ### 3. Data Flow ``` 1. Lobby Page GET /api/contests 2. Backend returns configured contests (from hardcoded config or Supabase) 3. Lobby verifies each contest has data GET /api/tracks/:track/entries?date=... 4. Backend checks Redis cache first If cached: Return immediately (no MongoDB call) If not cached: Query MongoDB Cache result Return 5. Player clicks "Enter" Navigate to matchups page ``` ## Benefits ### For Testing - **No repeated MongoDB calls** during testing sessions - Data cached for 24 hours (past data) or 1 hour (future data) - Fast response times after first load ### For Production - **Pre-fetch data** 1 day before contest starts - Admin can configure which contests to show - Reduce MongoDB usage/costs - Better user experience (faster loading) ## Setup Instructions ### 1. Redis (Optional but Recommended) 1. Sign up for [Upstash Redis](https://upstash.com/) (free tier) 2. Create a Redis database 3. Copy REST URL and Token 4. Add to `backend/.env`: ``` UPSTASH_REDIS_REST_URL=https://your-redis-instance.upstash.io UPSTASH_REDIS_REST_TOKEN=your_redis_token ``` **If you skip Redis**: The system will use in-memory caching (works fine for single-server deployments, but cache is lost on restart). ### 2. Contest Configuration (Current) Currently hardcoded in `backend/src/utils/contests.ts`: - 6 tracks for November 2, 2025 - All contests are active ### 3. Future: Supabase Integration When implementing admin panel, create this table in Supabase: ```sql CREATE TABLE contests ( id TEXT PRIMARY KEY, track TEXT NOT NULL, track_name TEXT NOT NULL, date DATE NOT NULL, is_active BOOLEAN DEFAULT true, created_at TIMESTAMP DEFAULT NOW(), updated_at TIMESTAMP DEFAULT NOW() ); ``` Then update `backend/src/utils/contests.ts` to query Supabase instead of using hardcoded data. ## Testing 1. **First Request**: MongoDB call + cache set 2. **Subsequent Requests**: Cache hit (no MongoDB call) 3. **After 24 hours** (past data) or **1 hour** (future data): Cache expires, new MongoDB call To test caching: ```bash # Make first request curl http://localhost:3001/api/tracks/GP/entries?date=2025-11-02 # Make second request (should be faster, check backend logs - no MongoDB query) curl http://localhost:3001/api/tracks/GP/entries?date=2025-11-02 ``` ## Cache Keys - `entries:TRACK:DATE` - Track entries data - `results:TRACK:DATE` - Track results data - `contests:active` - Active contests list - `contest:TRACK:DATE` - Specific contest config ## Next Steps 1.
