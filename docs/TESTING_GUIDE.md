# Complete Testing Guide ## Prerequisites Before testing, ensure: 1. **Backend is running**: ```bash cd backend npm run dev ``` Should see: ` Server running on http://localhost:3001` 2. **Frontend is running**: ```bash npm run dev ``` Should see: `Ready on http://localhost:3000` 3. **Cache is cleared** (if you want fresh start): ```bash npx tsx scripts/clear-cache.ts npx tsx scripts/clear-redis.ts ``` 4. **You're logged in as admin** (check navigation bar for "Admin" link) --- ## Test 1: Full Contest Creation Flow ### Step 1: Fetch Track Data **Where**: Admin Panel Contests Tab "Create New Contest" **What to do**: 1. Click on "Track" field 2. Type track name (e.g., "Gulfstream" or "GP") 3. Select from autocomplete dropdown 4. Select a date (e.g., `2025-11-02`) 5. Click "Fetch Track Data from MongoDB" **What to expect**: - Console shows: `[operationId] Starting track data fetch...` - Console shows: `[operationId] ️ Fetch completed in Xms` - Success message appears - **Track Preview** appears showing: - Track name (e.g., "Gulfstream Park (GP)") - Date formatted nicely - **Stats grid**: - Races count - Total Horses - Non-Scratched horses - **Jockeys count** (should be > 0) - **Trainers count** (should be > 0) - **Sires count** (should be > 0) - Post times (if available) **Expected time**: 5-15 seconds depending on data size --- ### Step 2: Create Contest **What to do**: 1. After preview appears, scroll down to "Contest Details" 2. Fill in: - **Contest Type**: Select from dropdown (Jockey vs Jockey, etc.) - **Entry Fee** (optional): e.g., `10` - **Prize Pool** (optional): e.g., `1000` 3. Click "Create Contest" **What to expect**: - Success message: "Contest created successfully!" - Form resets to "Fetch Track Data" step - Contest appears in "Manage Contests" table below **Check in database**: - Contest status should be `draft` - `matchup_types` should be empty array `[]` - `is_active` should be `false` --- ### Step 3: Set Matchup Types **What to do**: 1. In "Manage Contests" table, find your contest 2. Click the **Settings icon** (️) button 3. Modal opens: "Set Matchup Types" **What to expect**: - Modal has **static height** (doesn't jump around) - Checkboxes for: - Jockey vs Jockey (JvJ) - Trainer vs Trainer (TvT) - Sire vs Sire (SvS) - Mixed Matchups 4. Select at least 2 types (e.g., JvJ and TvT) 5. Click "Configure Settings" **What to expect**: - Modal switches to per-type settings view - Shows: "Step 1 of 2" (or however many you selected) - First matchup type name appears (e.g., "Configure Jockey vs Jockey (JvJ)") --- ### Step 4: Configure Per-Type Settings **What to do**: 1. **For JvJ** (first step): - Adjust sliders: - Matchups per Type: `10` (default) - Salary Tolerance: `500` (default) - Max Attempts: `500` (default) - 1v1 Preference: `80%` (default) - Click "Next" 2. **For TvT** (second step): - Adjust settings (can be different from JvJ) - Click "Next" 3. **Summary page**: - Should show all configured types with their settings - Should show **estimated calculation time** - Review settings - Click "Calculate Matchups" **What to expect**: - Modal height stays consistent (no jumping) - Each step shows current step number (e.g., "Step 1 of 2") - Settings are saved per type - Summary shows all settings clearly - Estimated time is shown (e.g., "Estimated total calculation time: 45 seconds") --- ### Step 5: Matchup Calculation **What to expect**: - Modal shows: "Calculating matchups... This may take a few moments." - Spinner appears - Console shows progress: ``` [operationId] Starting matchup calculation... [operationId] Calculating jockey_vs_jockey matchups... (1/2) [operationId] Generated X jockey_vs_jockey connections [operationId] ️ Calculation took Xms [operationId] Stored X jockey_vs_jockey matchups [operationId] Calculating trainer_vs_trainer matchups... (2/2) ... [operationId] Contest marked as ready [operationId] Total matchups calculated: X ``` **Expected time**: - 10 matchups per type: ~10-30 seconds - 20 matchups per type: ~20-60 seconds - Depends on data complexity **What to check**: - Modal automatically closes after completion - Success message appears - Contest status in table changes to "Ready" (green badge) - Matchup count is shown (e.g., "20 matchups") --- ### Step 6: Enable Contest Visibility **What to do**: 1. In "Manage Contests" table, find your contest 2. Check that: - Status shows "Ready" (green badge) - Matchup count is shown - **Toggle button** (eye icon) is now **enabled** (not grayed out) 3. Click the **Toggle button** (eye icon) **What to expect**: - Button changes from "Hide" (eye-off) to "Show" (eye) or vice versa - `is_active` in database changes to `true` - Success message: "Contest shown in lobby" or "Contest hidden from lobby" - Contest appears/disappears in lobby page **Before enabling**: Toggle should be **disabled** (grayed out) **After calculation**: Toggle should be **enabled** (clickable) --- ### Step 7: View in Lobby **What to do**: 1. Navigate to **Lobby** page (from navigation) 2. Look for your contest **What to expect**: - Contest appears in the table - Shows correct track name - Shows correct date - Shows entry count (horses) - "Enter" button is visible **If not authenticated**: - Contest is visible - "Enter" button says "Sign In" with login icon - Clicking redirects to `/login` **If authenticated**: - "Enter" button says "Enter" with arrow icon - Clicking redirects to `/matchups` --- ## Test 2: Navigation & Authentication ### Test 2a: Unauthenticated User **What to do**: 1. Sign out (if logged in) 2. Navigate through pages **What to expect**: - Navigation shows: **Home** and **Lobby** only (no Results) - Lobby shows contests (if any exist) - "Enter" button says "Sign In" - Clicking "Enter" redirects to `/login` - Results page shows: "Please sign in to view your matchup results" with "Sign In" button ### Test 2b: Authenticated User **What to do**: 1. Sign in as regular user 2. Navigate through pages **What to expect**: - Navigation shows: **Home**, **Lobby**, and **Results** - Results page shows user's rounds and bankroll - Lobby "Enter" button works ### Test 2c: Admin User **What to do**: 1. Sign in as admin 2. Navigate through pages **What to expect**: - Navigation shows: **Home**, **Lobby**, **Results**, and **Admin** - Admin badge appears next to email - Admin page accessible --- ## Test 3: Inactivity & Button Responsiveness **What to do**: 1. Log in as admin 2. Go to Admin page 3. **Wait 5+ minutes** without clicking anything 4. Try clicking buttons: - Toggle contest visibility - Delete contest - Settings button - Refresh button **What to expect**: - **All buttons should still work** (no unresponsiveness) - No "Session expired" errors - Console may show session refresh logs (this is normal) - Operations complete successfully **If buttons stop working**: - Check console for errors - Check if backend is still running - Try refreshing the page --- ## Test 4: Matchup Settings Modal **What to test**: 1. Open matchup types dialog 2. Switch between steps 3. Change modal content **What to expect**: - **Modal height stays consistent** (no jumping) - Height is fixed at ~600px minimum - Content scrolls inside modal if needed - No layout shifts when switching steps --- ## Test 5: Track Preview Stats **What to check after fetching track data**: **Stats should show**: - **Races**: Number > 0 - **Total Horses**: Number > 0 - **Non-Scratched**: Number <= Total Horses - **Jockeys**: Number > 0 (unique jockey count) - **Trainers**: Number > 0 (unique trainer count) - **Sires**: Number > 0 (unique sire count) **If any stat is 0 or missing**: - Check backend logs for errors - Verify MongoDB data has that information - Check `backend/src/utils/trackDataTransform.ts` calculation --- ## Test 6: Contest Toggle Behavior **Test scenarios**: ### Scenario A: Before Matchups Calculated - Contest status: `draft` or `processing` - **Expected**: Toggle button is **disabled** (grayed out) - **Expected**: Tooltip or visual indication that it's disabled ### Scenario B: After Matchups Calculated - Contest status: `ready` - **Expected**: Toggle button is **enabled** (clickable) - **Expected**: Clicking toggles visibility - **Expected**: Lobby reflects changes immediately --- ## Test 7: Error Handling ### Test 7a: Invalid Track/Date **What to do**: 1. Enter invalid track code (e.g., "XXX") 2. Select date with no data 3. Click "Fetch Track Data" **What to expect**: - Error message: "No entries found for this track and date" - Error is displayed clearly - Form remains usable ### Test 7b: Network Error **What to do**: 1. Stop backend server 2. Try to fetch track data **What to expect**: - Error message appears - User can retry - No crash or white screen --- ## Test 8: Per-Type Settings Summary **What to check**: 1.
