# Phase 3: Backend APIs - Implementation Summary ## Overview Phase 3 implements the backend APIs needed for the new contest creation flow. This includes track data fetching, contest management, matchup calculation setup, and operations tracking. ## Completed Endpoints ### 1. Track Data Management #### `POST /api/admin/tracks/fetch` - **Purpose**: Fetch track data from MongoDB, cache in Redis and Supabase, return preview - **Request Body**: ```json { "track": "GP", "date": "2025-11-02" } ``` - **Response**: ```json { "success": true, "track_data_id": "uuid", "preview": { "races_count": 10, "horses_count": 102, "non_scratched_count": 98, "track_code": "GP", "date": "2025-11-02", "fetched_at": "2025-01-27T..." } } ``` - **Features**: - Checks Supabase for existing data (avoids duplicate fetches) - Fetches from MongoDB if not cached - Caches in Redis (24h for past, 1h for future) - Stores in Supabase `track_data` table - Returns metadata preview #### `GET /api/admin/tracks/:track/:date/preview` - **Purpose**: Get preview metadata without fetching full data - **Response**: Same preview format as fetch endpoint - **Features**: Checks Supabase first, falls back to Redis cache ### 2. Enhanced Contest Creation #### `POST /api/admin/contests` (Updated) - **New Fields**: - `track_data_id` (optional): Links contest to track_data - `status`: Automatically set to `'draft'` - `matchup_types`: Empty array initially - `is_active`: Set to `false` (not visible until ready) - **Behavior**: - Validates `track_data_id` if provided - Creates contest in `draft` status - Not visible in lobby until status is `ready` and `is_active = true` ### 3. Contest Management #### `PUT /api/admin/contests/:id/matchup-types` - **Purpose**: Set matchup types and trigger calculation - **Request Body**: ```json { "matchup_types": ["jockey_vs_jockey", "trainer_vs_trainer"] } ``` - **Valid Types**: `jockey_vs_jockey`, `trainer_vs_trainer`, `sire_vs_sire`, `mixed` - **Behavior**: - Updates contest with matchup types - Sets status to `'processing'` - TODO: Triggers async matchup calculation (Phase 3.5) #### `GET /api/admin/contests/:id/status` - **Purpose**: Get contest status and calculation progress - **Response**: ```json { "success": true, "status": "processing", "matchup_types": ["jockey_vs_jockey"], "matchups_count": 0, "matchups_calculated_at": null, "has_track_data": true } ``` #### `PUT /api/admin/contests/:id/visibility` - **Purpose**: Toggle contest visibility (is_active) - **Request Body**: ```json { "is_active": true } ``` - **Validation**: Only allows `is_active = true` if status is `'ready'` - **Behavior**: Clears contests cache after update ### 4. Operations Tracking #### `POST /api/admin/operations` - **Purpose**: Create a new operation for tracking async jobs - **Request Body**: ```json { "operation_type": "fetch_track_data", "input_data": { "track": "GP", "date": "2025-11-02" } } ``` - **Response**: Returns created operation with ID #### `GET /api/admin/operations/:id` - **Purpose**: Get operation status by ID - **Response**: Full operation object with status, progress, etc. #### `GET /api/admin/operations` - **Purpose**: List user's operations (or all if admin) - **Query Params**: `status`, `operation_type`, `limit` - **Response**: Array of operations #### `PUT /api/admin/operations/:id` - **Purpose**: Update operation status/progress - **Request Body**: ```json { "status": "processing", "progress": 50, "result_data": {...}, "error_message": "..." } ``` ## New Utility Functions ### `backend/src/utils/trackDataTransform.ts` - **`transformRaceDocs()`**: Transforms MongoDB race documents to TrackData format - **`calculateMetadata()`**: Calculates preview metadata from records - **Shared**: Used by both public tracks API and admin track data API ## Database Integration All endpoints interact with Supabase tables created in Phase 2: - **`track_data`**: Stores full MongoDB data and metadata - **`contests`**: Enhanced with `status`, `track_data_id`, `matchup_types`, `matchups_calculated_at` - **`matchups`**: Will store calculated matchups (Phase 3.5) - **`operations`**: Tracks async operations ## Authentication All admin endpoints require: - Valid JWT token in `Authorization: Bearer <token>` header - User must have `is_admin = true` in `profiles` table - Middleware validates on each request ## Caching Strategy - **Redis (Upstash)**: - Track entries: 24h for past, 1h for future - Track results: 24h for past, 1h for future - Contests list: 5 minutes (reduced from 1 hour) - **Supabase**: - Permanent storage for track_data, contests, matchups - Metadata for fast previews ## Next Steps (Phase 3.5) The only remaining task is: - **Async Matchup Calculation Service**: - Background job that calculates matchups when types are set - Updates contest status to `'ready'` when complete - Stores matchups in `matchups` table - Updates operation tracking ## Testing To test the endpoints: 1. **Fetch Track Data**: ```bash curl -X POST http://localhost:3001/api/admin/tracks/fetch \ -H "Authorization: Bearer <admin_token>" \ -H "Content-Type: application/json" \ -d '{"track": "GP", "date": "2025-11-02"}' ``` 2. **Get Preview**: ```bash curl http://localhost:3001/api/admin/tracks/GP/2025-11-02/preview \ -H "Authorization: Bearer <admin_token>" ``` 3. **Create Contest with Track Data**: ```bash curl -X POST http://localhost:3001/api/admin/contests \ -H "Authorization: Bearer <admin_token>" \ -H "Content-Type: application/json" \ -d '{ "track": "GP", "date": "2025-11-02", "track_data_id": "<from_fetch_response>" }' ``` 4. **Set Matchup Types**: ```bash curl -X PUT http://localhost:3001/api/admin/contests/<contest_id>/matchup-types \ -H "Authorization: Bearer <admin_token>" \ -H "Content-Type: application/json" \ -d '{"matchup_types": ["jockey_vs_jockey", "trainer_vs_trainer"]}' ``` 5. **Check Status**: ```bash curl http://localhost:3001/api/admin/contests/<contest_id>/status \ -H "Authorization: Bearer <admin_token>" ``` ## Files Created/Modified ### New Files: -
