# Data Flow Explanation ## Ideal Scenario (Normal Day) ### How Data Flows in Production ``` ┌─────────────┐ │ User │ │ (Browser) │ └──────┬──────┘ │ │ 1. User clicks "Enter Contest" ▼ ┌─────────────────┐ │ Frontend │ │ (Next.js) │ │ localhost:3000 │ └──────┬───────────┘ │ │ 2. Request: GET /api/contests ▼ ┌─────────────────┐ │ Backend API │ │ (Express) │ │ localhost:3001 │ └──────┬───────────┘ │ │ 3. Check Redis Cache ▼ ┌─────────────────┐ │ Redis Cache │ │ (Upstash) │ └──────┬───────────┘ │ │ 4a. Cache HIT Return cached data (FAST! ) │ 4b. Cache MISS Continue to MongoDB ▼ ┌─────────────────┐ │ MongoDB │ │ (Atlas) │ └─────────────────┘ │ │ 5. Query race data │ 6. Return data │ ▼ ┌─────────────────┐ │ Backend │ │ (Express) │ └──────┬───────────┘ │ │ 7. Store in Redis (cache for next time) │ 8. Return to frontend ▼ ┌─────────────────┐ │ Frontend │ │ (Next.js) │ └──────┬───────────┘ │ │ 9. Display data to user ▼ ┌─────────────┐ │ User │ │ (Browser) │ └─────────────┘ ``` ### Step-by-Step Flow 1. **User Action**: User clicks "Enter Contest" on lobby page 2. **Frontend Request**: Frontend calls `GET /api/contests` to backend 3. **Backend Checks Cache**: Backend first checks Redis cache - **Cache HIT** (data exists): Return immediately (fast!) - **Cache MISS** (no data): Continue to MongoDB 4. **MongoDB Query**: Backend queries MongoDB for race data 5. **Cache Storage**: Backend stores result in Redis for future requests 6. **Response**: Backend returns data to frontend 7. **Display**: Frontend shows contests to user ### Why This is Efficient - **First Request**: MongoDB Redis Frontend (slower, ~500ms) - **Subsequent Requests**: Redis Frontend (fast, ~50ms) - **Cache Duration**: - Past races: 24 hours (won't change) - Future races: 1 hour (might have updates) --- ## What's Happening Right Now ### Current State ``` ┌─────────────┐ │ User │ │ (Browser) │ └──────┬──────┘ │ │ 1.
