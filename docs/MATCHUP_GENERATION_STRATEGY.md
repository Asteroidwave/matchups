# Matchup Generation Strategy ## Current Architecture ### Two Types of Matchup Generation 1. **Client-Side (Frontend/Browser)** - **Where**: Runs in the user's browser (their machine) - **When**: When user clicks "Reload" button or page loads - **Speed**: Very fast (<100ms) - just shuffles and regenerates from already-loaded connections - **Data**: Uses connections already loaded in memory - **Code**: `lib/matchups.ts` `generateMatchups()` function - **Limitation**: Can only generate from connections already loaded 2. **Backend (Server)** - **Where**: Runs on your backend server - **When**: When admin triggers matchup calculation for a contest - **Speed**: Slower (5-30 seconds) - fetches data, generates connections, calculates matchups - **Data**: Fetches from MongoDB, generates connections, stores in Supabase - **Code**: `backend/src/services/matchupCalculation.ts` - **Storage**: Matchups stored in Supabase `matchups` table ## What Happens When User Clicks "Reload"? ### Current Behavior (Client-Side) 1. User clicks "Reload" button 2. Frontend calls `regenerateMatchups()` 3. Function shuffles connections and regenerates matchups using same algorithm 4. Takes <100ms (very fast) 5. Shows new matchups immediately **Pros:** - Instant feedback - No server load - Works offline (once connections loaded) **Cons:** - Limited variety (only from loaded connections) - Same pool of connections = similar matchups - No guarantee of quality/balance ## Recommendations ### Option 1: Pre-Generate Many Matchups (Recommended) **Strategy:** - Generate **20-30 matchups per type** when admin calculates - Store all in Supabase - Frontend loads a random subset (e.g., 10) on page load - Reload button = load different random subset from pre-generated pool **Implementation:** ```typescript // Backend: Generate 30 matchups per type const matchups = generateUnifiedMatchups(connections, { ...settings, num_matchups: 30, // Generate more upfront }); setMatchups(shuffled); } else { // Slower: Generate new ones client-side const newMatchups = generateMatchups(connections, { tolerance }); setClientGeneratedMatchups(newMatchups); setMatchups(newMatchups); } }; ``` ## Reload Button Limits ### Recommended Limits 1. **Rate Limiting (Per Session)** - Max 10-15 reloads per session - Show countdown: "5 reloads remaining" - Reset on page refresh 2. **Cooldown Timer** - 2-3 second cooldown between reloads - Disable button during cooldown - Show spinner/countdown 3. **Visual Feedback** - Show "Generating..." spinner - Display reload count - Show time since last reload ### Implementation Example ```typescript const [reloadCount, setReloadCount] = useState(0); Refresh page to reset."); return; } // Reload setReloadCount(prev => prev + 1); setLastReloadTime(now); regenerateMatchups({ tolerance }); }; ``` ## Mixed Matchups Strategy **Mixed matchups** (JvJ + TvT + SvS combined) are most popular, so: 1. **Generate more upfront**: 40-50 mixed matchups (vs 20-30 for individual types) 2. **Show more on page**: 15-20 mixed matchups (vs 10 for individual types) 3. **Faster reload**: Pre-generated pool ensures instant reload 4. **Better variety**: Larger pool = more diverse matchups ## Final Recommendation **Use Option 3 (Hybrid) with these settings:** - **Backend**: Generate 30 matchups per type (JvJ, TvT, SvS), 50 for Mixed - **Frontend**: Load 10 random ones on page load (15 for Mixed) - **Reload**: Load different subset from pre-generated pool (fast) - **Limits**: Max 15 reloads per session, 2s cooldown - **Fallback**: If user wants more, allow client-side generation (with warning about speed) This gives: - Fast reloads (pre-generated pool) - Quality guaranteed (backend validation) - Variety (large pool) - Protection against spam (limits) - Flexibility (client-side fallback)
