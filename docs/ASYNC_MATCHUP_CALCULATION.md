# Async Matchup Calculation Service ## Overview The async matchup calculation service runs in the background when an admin sets matchup types for a contest. It calculates matchups for each type (JvJ, TvT, SvS) and stores them in Supabase. ## How It Works ### Flow 1. **Admin sets matchup types** `PUT /api/admin/contests/:id/matchup-types` 2. **Contest status updated** `status: 'processing'` 3. **Operation created** Tracks progress in `operations` table 4. **Async calculation starts** `calculateContestMatchups()` runs in background 5. **Matchups calculated** For each matchup type: - Load track data from Supabase or Redis - Generate connections (jockeys, trainers, sires) - Generate matchups (10 per type, tolerance: 500) - Store in `matchups` table 6. **Contest marked ready** `status: 'ready'`, `matchups_calculated_at` set 7. **Operation completed** `status: 'completed'`, progress: 100% ### Key Functions #### `calculateContestMatchups(contestId, matchupTypes, operationId?)` **Parameters:** - `contestId`: UUID of the contest - `matchupTypes`: Array of matchup types (`['jockey_vs_jockey', 'trainer_vs_trainer']`) - `operationId`: Optional operation ID for tracking **Returns:** ```typescript { success: boolean; matchupsCount: number; error?: string; } ``` **Process:** 1. Fetches contest from Supabase 2. Gets track data from `track_data` table or Redis cache 3. For each matchup type: - Generates connections (filters by role) - Generates matchups (10 per type) - Stores in `matchups` table 4. Updates contest status to `'ready'` 5. Updates operation tracking ### Matchup Generation Algorithm - **Pool**: All connections of the same role (jockey, trainer, or sire) - **Filtering**: Only connections with `apps >= 2` and `pointsSum > 0` - **Types**: - 1v1 (80% chance) - 2v1 or 1v2 (20% chance) - **Tolerance**: 500 (salary difference threshold) - **Count**: 10 matchups per type ### Storage - **Matchups Table**: One row per `(contest_id, matchup_type)` combination - **Matchup Data**: Full matchup objects stored as JSONB - **Format**: Same as frontend `Matchup` type ## Integration ### Endpoint Integration The service is automatically triggered when: ```typescript PUT /api/admin/contests/:id/matchup-types { "matchup_types": ["jockey_vs_jockey", "trainer_vs_trainer"] } ``` **Response:** ```json { "success": true, "contest": {...}, "operation_id": "uuid", "message": "Matchup types set. Calculation started in background." } ``` ### Operation Tracking Every calculation creates an operation record: - `operation_type`: `'calculate_matchups'` - `status`: `'processing'` `'completed'` or `'failed'` - `progress`: 0 100 - `input_data`: `{ contest_id, matchup_types }` - `result_data`: `{ matchups_count, contest_id }` ## Error Handling - **Track data not found**: Returns error, keeps status as `'processing'` (retry possible) - **Not enough connections**: Skips that matchup type, continues with others - **Calculation failure**: Updates operation to `'failed'`, keeps contest as `'processing'` - **Supabase errors**: Logs error, returns failure result ## Monitoring ### Check Calculation Status ```sql -- Get contest status SELECT id, status, matchup_types, matchups_calculated_at FROM contests WHERE id = '<contest_id>'; -- Get matchups count SELECT matchup_type, COUNT(*) as count FROM matchups WHERE contest_id = '<contest_id>' GROUP BY matchup_type; -- Get operation status SELECT * FROM operations WHERE input_data->>'contest_id' = '<contest_id>' ORDER BY created_at DESC LIMIT 1; ``` ### Backend Logs The service logs progress: ``` [calc-xxx] Starting matchup calculation for contest xxx [calc-xxx] Contest found: GP on 2025-11-02 [calc-xxx] Loaded track data from Supabase [calc-xxx] Processing 102 records [calc-xxx] Calculating jockey_vs_jockey matchups... [calc-xxx] Generated 45 jockey connections [calc-xxx] Generated 10 jockey_vs_jockey matchups [calc-xxx] Stored 10 jockey_vs_jockey matchups [calc-xxx] Contest xxx marked as ready [calc-xxx] Total matchups calculated: 20 ``` ## Testing ### Manual Test 1. Create contest with track data: ```bash POST /api/admin/tracks/fetch POST /api/admin/contests (with track_data_id) ``` 2. Set matchup types: ```bash PUT /api/admin/contests/<id>/matchup-types { "matchup_types": ["jockey_vs_jockey"] } ``` 3. Check status (poll every few seconds): ```bash GET /api/admin/contests/<id>/status ``` 4. Verify matchups in Supabase: ```sql SELECT * FROM matchups WHERE contest_id = '<id>'; ``` ## Performance Considerations - **Async**: Calculation runs in background, doesn't block API response - **Memory**: Processes one matchup type at a time to reduce memory usage - **Timeout**: No explicit timeout (relies on Node.js process limits) - **Concurrent**: Multiple calculations can run simultaneously (one per contest) ## Future Improvements - [ ] Add progress updates during calculation (update operation progress) - [ ] Add retry mechanism for failed calculations - [ ] Add cancellation support - [ ] Add calculation queue for better resource management - [ ] Add webhook/notification when calculation completes
