# New Contest Creation Flow - Architecture Plan ## Current Problems 1. Admin privileges lost on page reload 2. Lobby shows nothing inconsistently 3. Hard reload issues (sometimes works, sometimes doesn't) 4. No visibility into operations (what's happening? where did it fail?) 5. No handling for interruptions (logout during operations) 6. Contests appear immediately without data validation ## New Flow Requirements ### Phase 1: Data Fetching & Preview 1. Admin enters: Track Code + Date 2. **First call:** Fetch from MongoDB 3. **Cache data:** Store in Redis (Upstash) + Supabase `track_data` table 4. **Show preview:** - Number of races - Post times - Dates of races - Number of horses 5. **Admin confirms** Contest created (but NOT visible in lobby) ### Phase 2: Contest Creation 6. Contest appears in "Manage Contests" (status: "Draft" or "Processing") 7. Admin selects matchup types: - Jockey vs Jockey (JvJ) - Trainer vs Trainer (TvT) - Sire vs Sire (SvS) - Mixed (all types) 8. **Backend calculates matchups** (async, in background) 9. **Banner notification:** "Matchups ready for [Track] on [Date]" 10. Data stored: Supabase `matchups` table + Redis cache ### Phase 3: Visibility Toggle 11. Admin can toggle visibility (eye icon) 12. Only visible contests appear in lobby 13. Lobby shows contests with calculated matchups ## Robustness Requirements ### 1. State Management - Clear loading states ("Fetching data...", "Calculating matchups...", "Ready") - Progress indicators (progress bars, spinners) - Error states with retry buttons - Success confirmations ### 2. Error Handling - Network failures Retry with exponential backoff - Partial data Show what's available, allow retry - Timeouts Show timeout message, allow retry - Validation errors Clear error messages ### 3. Interruption Handling - Logout during operation Save progress, resume on login - Page reload Restore state from Supabase/Redis - Browser close Operations continue in background, resume on return ### 4. Data Persistence - Track data Supabase `track_data` table (permanent) - Contest config Supabase `contests` table - Matchups Supabase `matchups` table (permanent) - Cache Redis (Upstash) for fast access - Operations Supabase `operations` table (track progress) ### 5. Logging & Debugging - Backend: Structured logs (operation, step, status, error) - Frontend: Console logs with operation IDs - Network: Request/response logging - State: Log state changes (what changed, why) ### 6. Admin Session Persistence - Profile cached in localStorage (with expiration) - Session refresh on page load - Graceful degradation if refresh fails ## Database Schema Changes ### New Tables Needed #### 1. `track_data` table ```sql CREATE TABLE track_data ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), track_code TEXT NOT NULL, date DATE NOT NULL, data JSONB NOT NULL, -- Full MongoDB data metadata JSONB, -- races count, horses count, post times fetched_at TIMESTAMPTZ DEFAULT NOW(), UNIQUE(track_code, date) ); ``` #### 2. `matchups` table ```sql CREATE TABLE matchups ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), contest_id UUID REFERENCES contests(id), matchup_type TEXT NOT NULL, -- 'jockey_vs_jockey', 'trainer_vs_trainer', etc. matchup_data JSONB NOT NULL, -- Full matchup data calculated_at TIMESTAMPTZ DEFAULT NOW(), INDEX(contest_id, matchup_type) ); ``` #### 3. `operations` table (for tracking async operations) ```sql CREATE TABLE operations ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), user_id UUID REFERENCES auth.users(id), operation_type TEXT NOT NULL, -- 'fetch_track_data', 'calculate_matchups' status TEXT NOT NULL, -- 'pending', 'processing', 'completed', 'failed' progress INTEGER DEFAULT 0, -- 0-100 input_data JSONB, result_data JSONB, error_message TEXT, created_at TIMESTAMPTZ DEFAULT NOW(), updated_at TIMESTAMPTZ DEFAULT NOW() ); ``` #### 4. Update `contests` table ```sql ALTER TABLE contests ADD COLUMN IF NOT EXISTS status TEXT DEFAULT 'draft'; -- 'draft', 'processing', 'ready', 'active' ALTER TABLE contests ADD COLUMN IF NOT EXISTS track_data_id UUID REFERENCES track_data(id); ALTER TABLE contests ADD COLUMN IF NOT EXISTS matchup_types TEXT[]; -- ['jockey_vs_jockey', 'trainer_vs_trainer'] ALTER TABLE contests ADD COLUMN IF NOT EXISTS matchups_calculated_at TIMESTAMPTZ; ``` ## API Endpoints Needed ### Track Data - `POST /api/admin/tracks/fetch` - Fetch from MongoDB, cache, return preview - `GET /api/admin/tracks/:track/:date` - Get cached track data - `GET /api/admin/tracks/:track/:date/preview` - Get preview (metadata only) ### Contest Creation - `POST /api/admin/contests` - Create contest (with track_data_id) - `PUT /api/admin/contests/:id/matchup-types` - Set matchup types, trigger calculation - `GET /api/admin/contests/:id/status` - Get contest status (draft, processing, ready) - `PUT /api/admin/contests/:id/visibility` - Toggle visibility ### Matchups - `POST /api/admin/matchups/calculate` - Calculate matchups (async) - `GET /api/admin/matchups/:contest_id` - Get calculated matchups - `GET /api/admin/matchups/:contest_id/status` - Get calculation status ### Operations - `GET /api/admin/operations/:id` - Get operation status - `GET /api/admin/operations` - List user's operations ## Frontend Components Needed ### 1. `FetchTrackDataForm` - Input: Track code, Date - Button: "Fetch Data" - Shows: Loading Preview Confirm ### 2. `TrackDataPreview` - Shows: Races count, Post times, Horses count - Button: "Confirm & Create Contest" ### 3. `ContestConfiguration` - Shows: Contest details - Select: Matchup types (checkboxes) - Button: "Calculate Matchups" - Status: "Processing..." "Ready" ### 4. `OperationStatusBanner` - Shows: Current operation status - Auto-dismisses when complete - Shows progress bar ### 5. `EnhancedContestManagement` - Shows: Status badges (Draft, Processing, Ready) - Toggle: Visibility (only if Ready) - Actions: View preview, Recalculate matchups ## Implementation Plan ### Phase 1: Fix Current Robustness Issues (Priority 1) 1. Fix admin session persistence 2. Fix lobby loading inconsistency 3. Add comprehensive logging 4. Add error boundaries ### Phase 2: Database Schema (Priority 2) 1. Create new tables 2. Migrate existing data 3. Update RLS policies ### Phase 3: Backend APIs (Priority 3) 1. Track data fetching & caching 2. Contest creation with status 3. Matchup calculation (async) 4. Operation tracking ### Phase 4: Frontend Components (Priority 4) 1. New contest flow UI 2. Status indicators 3. Progress tracking 4. Error handling ### Phase 5: Testing & Polish (Priority 5) 1. Test all edge cases 2. Test interruptions 3. Performance optimization 4. Documentation ## Questions for Clarification 1. **Matchup calculation:** Should it happen automatically when matchup types are selected, or manually triggered? 2. **Data storage:** Should we store full MongoDB data in Supabase, or just metadata + cache in Redis? 3. **Matchup types:** Can a contest have multiple types simultaneously (e.g., both JvJ and TvT)? 4. **Preview:** What level of detail in preview? Just counts, or actual race schedules? 5. **Cache invalidation:** How long should cached data live? When should it be refreshed? ## Next Steps 1. **Immediate:** Fix admin session and lobby loading issues 2. **Short-term:** Create database schema 3. **Medium-term:** Implement new flow 4. **Long-term:** Optimize and polish
