# Tracks Table Analysis ## Current State The `tracks` table contains: - Track codes (GP, LRL, DMR, BAQ, CD, MNR) - Track names - `is_active` flag for visibility ## Purpose The `tracks` table is used for **track visibility management**: - Admins can toggle track visibility in the "Tracks" tab - This controls which tracks appear in the lobby - But this is **separate** from contest data ## Should We Keep It? ### Option 1: Keep Tracks Table (Recommended) - **Pros**: - Admins can control track visibility globally - Useful for enabling/disabling tracks without deleting contests - Clean separation of concerns - **Cons**: - Need to maintain track list - Extra table to manage ### Option 2: Remove Tracks Table - **Pros**: - Simpler - tracks are determined by active contests - Less data to manage - **Cons**: - Can't disable tracks without deleting contests - No way to hide tracks that have contests ## Recommendation **Keep the tracks table** but: 1. It should be **empty initially** 2. Tracks should be **added dynamically** when: - Admin fetches track data for first time - Or admin creates a contest for a new track 3. Use it for **visibility management only** ## Current Issue The tracks table has old test data. We should: 1. Clear it (or keep it if tracks are useful) 2. Make it dynamic - tracks added when contests are created ## Decision **Question for you**: Do you want to be able to hide/show tracks globally (separate from contests)? - **Yes** Keep tracks table, make it dynamic - **No** Remove tracks table, use contest data only
