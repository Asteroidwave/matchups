# Session Management & Caching Explained ## Your Question: "What does this affect - Upstash or Supabase?" Great question! Let me clarify what each service handles and what the session timeout affects. ## What Each Service Does ### Supabase (Authentication & Database) - **What it handles:** User authentication, JWT tokens, session management, database queries - **Session timeout:** **YES, this is what expires** ️ - Supabase JWT tokens expire after a default time (usually 1 hour) - When the token expires, API calls fail (buttons stop working) - This is why you see issues after ~5 minutes (Supabase auto-refreshes, but if that fails, buttons stop working) ### Upstash Redis (Caching) - **What it handles:** Cached data (track entries, contest lists, etc.) - **Session timeout:** **NO, not affected** - Redis cache is independent of user sessions - Cache TTL is set by us (currently 5 minutes for contests) - Cache doesn't care about user login status ## The Problem You Experienced ### What Was Happening: 1. **Supabase JWT token expires** (default ~1 hour, but can fail earlier) 2. **Supabase auto-refresh** sometimes fails silently 3. **Buttons stop working** because API calls fail authentication 4. **No visual feedback** - page looks fine but actions don't work ### Why It Happened: - Supabase's `autoRefreshToken: true` tries to refresh tokens automatically - But if the refresh fails (network issue, timing, etc.), the session becomes invalid - The app doesn't always detect this immediately - Result: "frozen" admin page where buttons don't work ## The Solution We Implemented ### 1. Proactive Session Refresh - **Refreshes 10 minutes BEFORE token expires** - Prevents the token from expiring while you're using the page - Uses `supabase.auth.refreshSession()` to get a new token ### 2. Auto-Reload After 1.5 Hours - **Safety checkpoint**: After 90 minutes, page automatically reloads - This ensures: - Fresh session - Latest data loaded - No stale state - Prevents very long sessions from accumulating issues ### 3. Session Duration Tracking - Tracks when you logged in - Automatically reloads after max duration (1.5 hours) - Reset when you log in again ## What This Affects ### Affected (Supabase): - **JWT token expiration** - Now refreshed proactively - **Admin API calls** - Will work for 1.5 hours continuously - **Button functionality** - Won't freeze after 5 minutes - **Session validity** - Maintained for full 1.5 hours ### NOT Affected (Upstash): - **Redis cache** - Completely independent - **Cache TTL** - Still 5 minutes for contests (as we set) - **Cached data** - Not affected by session timeout ## Configuration ### Current Settings (Admin Page): ```typescript useSessionRefresh({ refreshBeforeExpiry: 10 * 60 * 1000, // 10 minutes before token expires maxSessionDuration: 90 * 60 * 1000, // 1.5 hours (90 minutes) enableAutoReload: true, // Auto-reload after 1.5 hours adminOnly: true, // Only on admin pages }); ``` ### Can You Change It? **Yes!** You can adjust: - **`maxSessionDuration`**: Change from 90 minutes to any duration - Shorter (e.g., 60 min): More frequent checkpoints, more secure - Longer (e.g., 2 hours): Less interruption, but longer sessions - **`refreshBeforeExpiry`**: When to refresh (default: 10 min before) - Increase if you want more buffer (e.g., 15 min) - Decrease if you want less frequent refreshes (e.g., 5 min) ### Recommended Settings: - **1.5 hours (90 min)**: Good balance for admin work - **2 hours (120 min)**: If you need longer sessions - **1 hour (60 min)**: More secure, more frequent checkpoints ## How It Works ### Timeline Example (1.5 hour session): ``` 0:00 - Login 0:50 - First refresh (10 min before 1:00 expiry) 1:00 - Token would have expired, but already refreshed 1:20 - Second refresh (10 min before 1:30 expiry) 1:30 - Token would have expired, but already refreshed 1:30 - Auto-reload checkpoint (1.5 hours reached) Page reloads Fresh session New 1.5 hour timer starts ``` ### What Happens on Reload: 1. Page reloads 2. AuthContext checks for existing session 3. If session exists, continues (no re-login needed) 4. If session expired, redirects to login 5. New 1.5 hour timer starts ## Benefits ### What You Get: 1. **No more frozen buttons** - Session stays valid 2. **Seamless experience** - Can work for 1.5 hours without interruption 3. **Automatic checkpoint** - Page reloads to refresh everything 4. **No manual refresh needed** - Everything is automatic 5. **Better error recovery** - If something fails, reload happens ### ️ Trade-offs: 1. **Page reload after 1.5 hours** - Brief interruption (but everything saved) 2. **Slight Supabase API usage** - Refresh calls (negligible) 3. **Memory** - Slightly more due to session tracking (negligible) ## Summary | Component | What It Does | Session Timeout Affects? | |-----------|-------------|-------------------------| | **Supabase Auth** | User login, JWT tokens | **YES** - This is what we fixed | | **Supabase Database** | Store data (contests, profiles) | **YES** - Needs valid session to query | | **Upstash Redis** | Cache data (contests, tracks) | **NO** - Independent of sessions | | **Backend API** | Process requests | **YES** - Needs valid JWT token | **Bottom Line:** The session timeout affects **Supabase** (authentication), not **Upstash** (caching). We fixed the Supabase session issue so you can work for 1.5 hours without interruption, then the page auto-reloads as a safety checkpoint. ## Questions? ### Q: Why 1.5 hours specifically? **A:** Good balance between: - Long enough for extended admin work - Short enough to prevent stale sessions - Can be adjusted if needed ### Q: What if I'm in the middle of something when it reloads? **A:** - All data is saved to Supabase immediately - Reload happens after 1.5 hours (plenty of time) - You can adjust the duration if needed - Can disable auto-reload if you prefer (but then might hit session issues) ### Q: Does this use more Supabase quota? **A:** Minimal - refresh calls are lightweight (just token refresh, not full login). Well within free tier limits. ### Q: Can I disable auto-reload? **A:** Yes, set `enableAutoReload: false` in the hook, but then you might hit the original session expiration issue.
